## 1. Scope

**Concepts:** `var` function scope, hoisting, `let`/`const` block scope, output prediction.

| File | What it shows |
|------|---------------|
| [1. scope-var.js](1.%20scope/1.%20scope-var.js) | `var` is function-scoped ‚Äî block re-declaration overwrites the function-level variable |
| [2. hoisting.js](1.%20scope/2.%20hoisting.js) | `var` declarations are hoisted as `undefined`; the assignment stays in place |
| [3. scope-const-let.js](1.%20scope/3.%20scope-const-let.js) | `let`/`const` create a new binding per block ‚Äî no leakage |
| [4. guess/1. snippet.js](1.%20scope/4.%20guess/1.%20snippet.js) | `var` can be re-declared in the same scope; `let`/`const` cannot |
| [4. guess/2. snippet.js](1.%20scope/4.%20guess/2.%20snippet.js) | Hoisting inside a function ‚Äî prints `undefined` before assignment |
| [4. guess/3. snippet.js](1.%20scope/4.%20guess/3.%20snippet.js) | `var` leaks out of a block; `let` is confined to it |
| [4. guess/4. snippets.js](1.%20scope/4.%20guess/4.%20snippets.js) | `const` mutation throws `TypeError`; IIFE creates a private scope |

## 10. SOLID Principles

Each file shows a **bad** implementation and a **good** implementation side by side.

| File | Principle |
|------|-----------|
| [1. s.js](10.%20principles/solid/1.%20s.js) | **S** ‚Äî Single Responsibility: one class, one reason to change |
| [2. o.js](10.%20principles/solid/2.%20o.js) | **O** ‚Äî Open/Closed: add new behaviour by adding classes, not editing existing ones |
| [3. l.js](10.%20principles/solid/3.%20l.js) | **L** ‚Äî Liskov Substitution: subclasses must not break the parent's contract |
| [4. i.js](10.%20principles/solid/4.%20i.js) | **I** ‚Äî Interface Segregation: don't force classes to implement methods they don't use |
| [5. d.js](10.%20principles/solid/5.%20d.js) | **D** ‚Äî Dependency Inversion: inject abstractions; don't hard-wire implementations |
